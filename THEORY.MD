# Theory: One-Click Local Video Downloading

## Problem

Cat Catch proves that a browser extension can observe enough network traffic to find direct media, HLS manifests, DASH manifests, and separate audio/video tracks. Its interface exposes those resources because it is a general-purpose inspection tool. The personal downloader needs the opposite abstraction: one decision from the user, followed by automatic selection, downloading, merging, and naming.

The hard part is not the button. Browser media pipelines split responsibility across page requests, short-lived Manifest V3 service workers, browser-managed downloads, and resource-intensive media processing. Treating every detected URL as equivalent either floods the interface with fragments or selects a low-quality rendition. Treating every download as an FFmpeg job adds needless latency to direct MP4 files.

## Operating Theory

The extension should be a detector and orchestrator, not a media processor. It observes response metadata using `webRequest`, reduces noisy requests into typed candidates, and derives one plan per tab. A direct MP4 is already in the desired form and belongs on Chrome's native download path. HLS, DASH, and separate audio/video tracks require a native FFmpeg host because native processing streams large inputs without copying complete media through a browser WebAssembly filesystem.

“Highest compatible quality” is a selection policy shared by both layers. The extension prefers a master manifest over fragments and otherwise scores direct candidates using available quality and size hints. For adaptive manifests, FFmpeg's stream selection and muxing operate on the complete presentation, where resolution and audio streams can be evaluated correctly. The output path is always local, and successful jobs end as MP4 files in Downloads.

The native boundary carries control data only: URLs, an allowlisted subset of request headers, the desired job type, and progress events. Media bytes never cross Native Messaging. A narrowly bounded in-memory HLS manifest is the one exception: the extension may pass its text to the host, which resolves remote references and creates a private temporary playlist. The host invokes FFmpeg or an optional pinned yt-dlp module with argument arrays rather than shell commands, writes temporary output, and promotes it only after validation. This makes partial downloads controllable and prevents command injection through captured URLs or page titles.

Normal detection still does not need a content script. Correlating `onBeforeSendHeaders` with `onHeadersReceived` captures authorized request context and response MIME metadata without injecting code into the page. Deep Blob detection is an explicit, origin-scoped fallback: it is installed only after normal observation fails and the user grants Chrome's optional scripting permission. A persistent Native Messaging port owns native process lifetime and keeps progress reporting separate from the popup, so closing the popup does not cancel work.

## Strategy

The product keeps one primary action while using a layered fallback strategy. Direct MP4 remains browser-managed. Observed adaptive manifests and paired tracks go to FFmpeg. When observation has no candidate, a pinned, explicitly installed yt-dlp module may interpret the page URL; cookie transfer remains off unless the user grants the cookie permission for that action. Deep Blob detection is the final opt-in detector for pages that construct HLS entirely in memory. Each layer is narrower and more privileged than the previous one, so ordinary downloads keep the smallest possible boundary.

Native execution is organized as a responsive controller rather than a blocking function. The read loop owns a registry of job controllers while worker threads own FFmpeg or yt-dlp processes. Cancel means discard a finite partial file; stop means ask a live process to finalize, then promote the result only if ffprobe finds playable media. Bounded process termination, input reconnects, and whole-job retries are separate mechanisms: reconnects cover short transport interruptions, while full retries cover only classified transient failures and never authentication or malformed-input errors.

Chrome privileges follow the same lease model as native processes. Cookie and scripting access are requested only from a popup click, while the power permission is optional and represented as a reference-counted lease across active native jobs. Terminal events, disconnects, and worker recovery all converge on lease release. This is safer than scattering `requestKeepAwake` calls through UI branches and makes overlapping-job behavior testable without Chrome.

Codec choice is proactive rather than an all-or-nothing error fallback. ffprobe supplies the selected video and audio codecs; the MP4 policy chooses copy or transcode independently. A compatible H.264 video therefore remains byte-for-byte copied when an Opus audio stream must become AAC. A conservative full H.264/AAC pass remains a last resort for incomplete metadata or muxer edge cases.

Installation is intentionally a one-time personal setup. The unpacked extension exposes its generated ID; an installer registers a host manifest restricted to that exact origin. Both development and production use explicit `.venv` directories. The production installer creates a dependency-free virtual environment beside the copied host and registers a launcher bound to that interpreter, preventing Chrome from depending on whichever global Python happens to be first on `PATH`. This trades public-store portability for reliable native FFmpeg behavior, which matches the personal-only constraint.

After the first released and manually verified version, compatibility work follows a stricter hierarchy. Correctness failures come first: the declared browser floor must match the APIs that keep native jobs alive, candidates must remain inside one document/playback session, and FFmpeg must receive only options supported by its installed build. Common coverage gaps such as generic MIME responses and multi-language audio come next. HDR policy, child-frame injection, and signed-URL reacquisition remain conditional because each increases complexity or privilege and needs a failing fixture first.

The compatibility campaign is fixture-led. A local `.venv` server and locally generated FFmpeg media should reproduce SPA transitions, ads, two players, ambiguous response metadata, redirects, split tracks, adaptive/live streams, language variants, and optional Blob/extractor paths. This turns future fixes into falsifiable behavior rather than accumulating site-specific guesses or depending on third-party pages that change independently.

Project documentation should mirror the product's one-click abstraction without hiding its technical boundaries. The README therefore keeps its written feature and setup guidance as the durable source of truth, adds the brand board as the identity reference, and leads with a short silent workflow reel. The reel is authored as a deterministic, local-assets-only Hyperframes project under `hyperframes/readme-showcase`; its poster links to the MP4 so the repository still has a useful hero when inline video playback is unavailable.

## Key Discoveries

The original assumption that bundled WebAssembly best preserved a simple experience did not survive the large-file requirement. ffmpeg.wasm stages inputs and outputs through an in-browser filesystem and remains materially slower than native FFmpeg. Native Messaging adds installation work, but that cost is paid once and does not leak into the download interaction.

Cat Catch's useful leverage point is its two-stage observation pattern: retain selected request headers when a request is sent, then classify the resource when response headers reveal MIME type and size. The new implementation keeps that pattern while discarding the general resource browser, custom regex system, capture scripts, and online conversion UI.

The clean implementation also reveals a useful fallback boundary. Direct MP4 first uses `chrome.downloads`, preserving browser cookies and avoiding an FFmpeg process. If Chrome rejects that download, the same candidate can be remuxed by the native host. Adaptive and split inputs always use the host. FFprobe now chooses copy or transcode per selected stream before FFmpeg starts, with error-driven full transcoding retained only as a conservative fallback.

Verification confirms the native half of this theory. The suite creates a paced video/audio process, interrupts it through the same process-group signal used by live stop, and verifies that FFmpeg finalized both streams without force termination. The localhost smoke test recovers from an injected HTTP 503, downloads an inline HLS manifest with relative segments, merges separate tracks, and proves both selective paths: H.264 stays copied while Opus becomes AAC, and AAC stays copied while VP9 becomes H.264. A separate installation test creates a production `.venv`, launches the generated wrapper, and completes a framed capability ping. FFmpeg's documentation confirms that omitting `-map` for a manifest selects the highest-resolution video automatically, which makes the adaptive path's quality rule explicit rather than accidental.

A broader reference audit identifies process control as the enabling change. M3u8Downloader_H and FFandown demonstrate that retries, cancellation, live-stream state, and partial recovery matter more to reliability than another parser UI. A synchronous host cannot receive a stop message during FFmpeg work, so live capture, bounded cancellation, retry feedback, and page-extractor fallback all require the same worker/process registry. This makes concurrency control infrastructure, not another media parser, the central lever for the second release.

The same audit identifies two compatibility improvements that preserve the one-click theory. FFprobe can return codecs as well as duration, allowing the host to copy compatible video while transcoding only an incompatible audio stream. Elephant and Media Downloader show that yt-dlp is useful only at the detector's blind spot, not as a replacement for the fast normal paths. The fallback is therefore pinned to stable version `2026.6.9` inside the production `.venv`, loads no user configuration or plugin directories, and receives only the page URL plus explicitly opted-in cookies. Remote component loading, silent updates, REST servers, and general engine-management UIs stay outside the trusted boundary.

Implementation confirms that retry policy must account for the user's installed FFmpeg rather than only current upstream documentation. FFmpeg 6.1 supports the core reconnect, network-error, HTTP-status, streamed-input, timeout, and delay-limit options, but not newer global retry-count flags. The portable policy uses that common protocol subset and adds an application-level three-attempt exponential retry. This bounds behavior on older supported installations while keeping `reconnect_at_eof` exclusive to live inputs.

The control and permission seams are now independently verifiable. Twenty-six automated tests exercise an active worker receiving cancellation, both Windows kill and POSIX process-group kill fallbacks, bounded whole-job retries, independent codec decisions, private cookie files, relative HLS resolution, dependency preflight, an HLS-typed Blob fixture, and reference-counted wake locks. This reinforces the strategy of keeping orchestration rules pure around thin browser and subprocess boundaries.

The post-release audit exposes three compatibility assumptions that are currently too broad. The manifest claims Chrome 102 support even though `connectNative()` only gained service-worker keepalive guarantees in Chrome 105; planned document IDs require Chrome 106. Candidate selection is tab-scoped, so SPA navigation and multiple players can cross-contaminate selection or split-track pairing. Native HTTP flags are static even though FFmpeg protocol capabilities vary by build. These are higher priority than adding formats because they can disconnect a valid job, select the wrong media, or reject an otherwise usable installation.

Two coverage gaps are common enough to plan but not urgent enough to bypass fixtures. Direct files served through XHR with `application/octet-stream` are invisible to the current conservative classifier, and FFmpeg's default audio policy can choose a high-channel commentary or foreign-language track over the intended default. Both can be fixed with bounded response-confidence scoring and explicit stream intent without changing the one-click interface.

The showcase has one narrative job: make the local pipeline legible in twelve silent seconds. Its four beats are brand hook, automatic detection, contained FFmpeg merge, and local MP4 save. A recurring blue route line connects those beats, while exact brand colors and the existing SVG mark avoid inventing a second visual identity. Brand fidelity is context-aware rather than mechanically identical: the standard multicolor mark belongs on light surfaces, while an exact-geometry white variant belongs inside saturated blue processor states. Product mockups follow the same clarity rule—functional frames remain level and centered unless rotation carries real meaning. This is intentionally a stylized product explanation rather than a literal browser recording, because the browser UI can change while the product promise remains stable.

The completed motion asset validates this documentation model at repository scale: a 1920×1080 H.264 poster-linked MP4 is small enough for versioned project documentation, remains understandable without audio, and preserves the final saved-file frame when motion cannot play. The editable Hyperframes source stays separate from generated README assets, while both use only project-local artwork at capture time. Automated overflow and contrast checks remain necessary but insufficient for motion polish; semantic frame inspection at the center of each beat catches issues such as a deliberately tilted frame reading as misalignment or a correct brand mark disappearing against a same-hue processor surface.

## Open Questions

Authenticated sites vary in whether a copied Cookie, Referer, Origin, and User-Agent are sufficient for FFmpeg to retrieve a manifest. The first version allowlists those headers, but some sites may use expiring URLs or browser-only challenge state. DRM-protected streams remain explicitly unsupported.

Chrome cannot be exercised in the current WSL environment because no GUI browser is installed. Pure selection tests, Blob/power mocks, native FFmpeg integration, and the installed-host protocol test provide the automated evidence. The user has now reinstalled and reloaded release `0.2.0` in Windows Chrome and reports the extension working normally, closing the primary browser/native integration uncertainty.

Host placement across WSL and Windows is now explicit: registration happens in the operating system that runs the browser, and the installer creates the production `.venv` there. A Windows Chrome profile cannot see a host registered inside WSL, so the documented WSL command invokes Windows Python through `py.exe` and passes it the Windows form of the installer path.

The remaining uncertainty is concentrated in uncommon paths rather than the core architecture: optional permission denial, Windows process interruption, an actual yt-dlp-supported page, and a page that creates HLS inside a child-frame Blob. The next leverage point is reproducible end-to-end browser fixtures followed by browser-floor, candidate-scoping, and FFmpeg-capability corrections—not another downloader engine. Evidence that would change the architecture is a site whose authenticated media cannot be replayed by FFmpeg using the allowlisted headers; that would require a narrower browser-side bridge or site adapter rather than a broader native protocol.

Live classification is necessarily heuristic. A manifest without a finite ffprobe duration is treated as live so the popup offers “Stop and save,” but some event streams or unusual VOD manifests may also omit duration. Output validation prevents a bad partial file from being presented as complete; real authorized live fixtures are still needed to tune classification across HLS variants.

Deep Blob detection introduces a constrained page-world bridge. The detector sees only HLS-typed Blobs below a strict size limit, and it is registered only for an origin the user explicitly enables. The page already owns the manifest content; the security goal is to prevent that content from becoming arbitrary local-file access or unbounded Native Messaging input. Relative segment and key references must resolve to HTTP(S), and the host—not the page—creates and deletes the temporary playlist.
